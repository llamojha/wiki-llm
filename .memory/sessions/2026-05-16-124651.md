## Summary

Debugged and refactored the Vaultmark curation pipeline from a broken monolithic streaming endpoint into a client-driven multi-step flow, then designed a Karpathy-style wiki-llm spec for a Lambda-based single-call-per-source architecture. Fixed multiple issues along the way: 400 errors, IAM permissions, Vercel timeouts, perpetual spinners, and model ID misconfigurations.

## Key Decisions

- **Raw files are immutable** — never deleted after processing. `_processed.json` manifest tracks what's been ingested (hash-based change detection for re-processing).
- **Single Bedrock call per source (Karpathy style)** — outputs all affected files (pages + index + overview + log) atomically. Wiki is always coherent after each file.
- **Lambda for curation** — offloads the long-running Bedrock calls from Vercel's 10s Hobby timeout. Vercel triggers + polls, Lambda does the work.
- **Reindex = lint** — the reindex pass serves as Karpathy's lint (contradictions, orphans, stale claims, cross-ref gaps) and also indexes user-authored `wiki/` pages.
- **Spaces are user-controlled boundaries; LLM decides folder structure within them.**
- **Sequential processing within a space** (not parallel) for compounding — each file sees the wiki state from prior ingests.
- **Model:** `eu.amazon.nova-2-lite-v1:0` via Bedrock cross-region inference profile, region `eu-central-1`.
- **Human-in-the-loop ingest** noted as future goal (Karpathy's interactive style).
- **Ask-wiki (Phase 5)** should file good answers back into the wiki.
